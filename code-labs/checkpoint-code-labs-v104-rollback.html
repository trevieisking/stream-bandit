Current working state

Code Labs V104

Working. Reads Code Labs context, reads Code Labs URLs, and saves safe branch/PR write requests into Supabase SQL.

GitHub connector

Actual PR writer. GitHub creates branches, writes files, and opens pull requests. Code Labs does not directly write GitHub.

Supabase

Storage and history. Supabase holds Code Labs repair history, write-request queue, and the V104 MCP Edge Function.

Browser Code Labs pages

Helper pages only. They prepare handoffs, copy packets, track links, and save local/Supabase history. They do not hold a GitHub token.

What each tool is for

Code Labs V104

  • get_code_labs_context reads latest Code Labs project, job, packets, tests, and audit history.
  • read_code_labs_url reads public pages under /code-labs/ only.
  • save_code_labs_write_request saves one safe database-only write request for a branch/PR. It does not edit GitHub and does not open a PR.

GitHub

  • Creates the branch from main.
  • Creates or updates the exact requested file only.
  • Opens the pull request and returns the PR/review link.
  • Does not write to main directly unless Trevor explicitly asks after testing.

Supabase

  • Hosts the code-labs-mcp-stub Edge Function.
  • Stores Code Labs repair history and queued write requests.
  • Should not be used to edit Stream Bandit public files directly.

What ChatGPT can and cannot do

Can do

  • Read Code Labs context.
  • Read Code Labs public helper pages.
  • Save a safe write request in Code Labs SQL.
  • Use GitHub connector to create a branch and PR when GitHub is available.
  • Explain plain-English steps for non-coders.

Cannot do through Code Labs V104

  • Cannot directly write GitHub files.
  • Cannot open a PR by itself.
  • Cannot merge to main.
  • Cannot delete files unless a separate GitHub workflow is explicitly approved.
  • Cannot bypass missing full file contents.

Safe workflow from this checkpoint

  1. Use Code Labs V104 to read context or save a write request.
  2. Use GitHub to create a branch from main.
  3. Touch only the exact target file.
  4. Open a PR. Do not merge.
  5. Use GitHub Tracker to store PR and preview links.
  6. Trevor tests the preview.
  7. Only after Trevor approves, continue to merge or promote.
Hard rule: no more help-tool rewrites unless Trevor explicitly asks. Documentation/checkpoint files are allowed only as branch/PR work.
